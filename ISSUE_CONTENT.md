**「核心开源 vs 企业增值」的技术分层策略**：

| 层级 | 开源版 (Community) | 企业版 (Enterprise) | 技术实现方式 |
|-----|-------------------|-------------------|-------------|
| **核心引擎** | ✅ 完全开源 | ✅ 完全开源 | 同一代码库，MIT/Apache 协议 |
| **基础插件** | ✅ 开源 | ✅ 开源 | 官方维护，社区共建 |

| **管理后台** | 基础版 | 企业增强版 | 特性开关 (Feature Toggle) |
| **技术支持** | 社区支持 | SLA 保障 | 独立工单系统 |

**开源友好的架构设计原则**：

1. **模块化微内核架构**：
   - 核心引擎保持轻量，只提供基础能力
   - 所有功能通过插件扩展，插件可独立开发、独立发布
   - 插件接口公开文档化，降低社区贡献门槛

2. **Feature Toggle 机制**：
   ```yaml
   # config.yaml
   features:
     basic_workflow: true          # 开源版可用
     advanced_analytics: false     # 开源版禁用，企业版启用
     custom_branding: false        # 企业版专属
   ```
   通过 license 文件控制特性开关，同一套代码运行不同版本

3. **API 分层设计**：
   - **Public API**: 稳定、向后兼容，对社区承诺长期支持
   - **Internal API**: 内部使用，可能变更
   - **Enterprise API**: 企业版专属能力
